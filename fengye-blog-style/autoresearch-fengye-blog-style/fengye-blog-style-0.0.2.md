# Fengye 博客文风

## 概览

这个 skill 用于中文技术博客写作，目标是尽量贴近用户在博客站点里的既有文风。

开始写之前，先看用户已有博客，而不是只凭当前草稿猜文风。优先参考这些在线入口：

- 博客首页：[https://fengye404.top/](https://fengye404.top/)
- 归档页：[https://fengye404.top/archives/](https://fengye404.top/archives/)

目标文风大致是：

- 解释清楚，但不软、不说教
- 技术判断站得住，笔锋更硬一些
- 更像技术分享，不像营销文案
- 愿意解释，但不会为了显得“有思考”而故意绕

## 写作目标

文章通常应该呈现为：

1. 先把问题说清楚
2. 再把工程判断讲明白
3. 然后给出具体方案、改动或结构
4. 最后落到实际结论和启发

不要追求“文艺感”或者“很会写”的感觉，更重要的是可读性、信息密度和工程可信度。

## 核心规则

- 要解释清楚，但不要为了压字数过度简化。用户是希望“讲明白”的。
- 行文要直接，优先用陈述句，不要靠很多铺垫进入正题。
- 尽早使用具体工程名词，比如：模块边界、验证、CI、schema、契约、invariants、runtime、migration。
- 如果标题里出现项目名，开头要尽快用一两句说明这个项目是什么，不要让读者一直悬着。
- 如果作者背景需要交代，只能自然带一句，不能写得像在辩解或自我解释。
- 外部资料要内化，不要把文章写成“文献综述”，除非用户明确要求。
- 在重写或大幅润色之前，先从归档页抽样阅读多篇旧文。
  普通润色至少看 3 到 5 篇；如果要重写整篇结构或重新找文风，优先看 8 到 10 篇。
- 参考旧文时，重点观察这些东西：开头怎么起笔、问题怎么落地、解释密度怎么控制、结尾怎么收，不要只模仿表面句式。

## 反模式

以下写法**绝对禁止**，不存在"除非必要"的例外：

- **禁止** `不是……而是……` 句式。一次都不能出现。如果想对比两个概念，直接说后者是什么，不要先否定前者。例如：❌ "这不是一个简单的工具，而是一个完整的平台" → ✅ "这是一个完整的平台"
- **禁止** `本文不讨论……只讨论……`。直接说讨论什么。
- **禁止** `换句话说`。要么前一句话已经说清楚了（那就删掉"换句话说"后面的内容），要么没说清楚（那就重写前一句）。
- **禁止** 防御式表述："我不是在做 X，我是在做 Y"。直接说在做 Y。
- **禁止** `在某种程度上`、`或许`、`可能是因为`、`某种意义上说` 等模糊表达。要么确认了写断言，要么不确认就不写。
- 过多口语填充
- 一直重复解释同一个点的“AI 说明文”语气
- 用“它不是什么，而是什么”来介绍项目
- 在正文里堆很多外部参考链接，像综述一样

一个判断标准是：如果一句话像是在努力证明自己“很有思考”，就应该继续简化。

## 结构建议

大多数技术文章，优先按这个结构来：

### 引言

- 先定义主题
- 说清楚这篇文章为什么要讨论它
- 用真实项目或工程背景把文章钉住

### 问题

- 明确点出工程问题
- 把边界说清楚

### 方法

- 解释具体改动、约束、判断
- 有必要时用具体例子

### 结论

- 总结工程上的启发
- 如果合适，可以给 checklist、项目链接或后续方向

## 工作方式

当使用这个 skill 时，应该这样工作：

1. 先看博客首页和归档页，再抽样阅读多篇旧文建立语感
2. 再读当前草稿
3. 找出语气、结构、framing 不匹配的地方
4. 必要时整体重写，不要只修补几句话
5. 保留有价值的技术细节
6. 如果用户要求发布，就继续完成：改文章、构建站点、部署、回报 commit id
