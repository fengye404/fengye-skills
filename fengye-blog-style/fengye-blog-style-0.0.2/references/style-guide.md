# Fengye 博客文风补充说明

## 风格总结

用户想要的博客文风大致是：

- 中文技术写作
- 解释要足够，让读者真正看明白
- 但不能软、不能飘、不能像在做演讲
- 更像"技术分享"，不像"项目回顾感想"
- 更像"工程判断"，不像"概念布道"

在判断文风之前，先回到用户已经公开发布的博客去看，不要只盯着当前这一篇。优先从这里进入：

- 博客首页：[https://fengye404.top/](https://fengye404.top/)
- 归档页：[https://fengye404.top/archives/](https://fengye404.top/archives/)

如果只是小幅润色，先抽样看 3 到 5 篇。
如果是整篇重写、重新找语感，先看 8 到 10 篇，再下笔。

## 这里说的"更硬"是什么意思

"更硬"不是一味更短。

它意味着：

- 少一点进入正题前的铺垫
- 少一点绕来绕去的修辞
- 少一点情绪化的过渡句
- 多一点直接的问题陈述
- 多一点精确的工程语言

## 应该保留的东西

- 清晰的因果解释
- 具体例子
- 显式的问题拆解
- 最后能够回到真实工程工作的结论

## 应该删掉的东西

- 反复使用 `不是……而是……`
- 过多元叙述
- AI 味很重的抽象套抽象
- "这个很有意思，因为……"这类空洞铺垫
- 过多自我定位和自我解释
- 过度推销项目

## 引言检查清单

在定稿引言之前，检查：

1. 读者知道文中项目/工具是什么吗？
2. 项目有没有在开头一两句里说明白？
3. 文章的真正主题清楚吗？
4. 引言里有没有多余的自我解释？

## 技术分享检查清单

在定稿整篇文章之前，检查：

1. 核心工程问题是否说清楚了？
2. 具体改动或约束是否讲明白了？
3. 思想和实现有没有分清？
4. 结尾有没有把读者该带走的东西说出来？
5. 如果文章用了项目案例，这个项目是在服务论点，还是抢走了文章主题？
6. 这篇文章的语气，和博客站里已有文章放在一起读时，会不会显得太像 AI 代笔？

## 发布到 fengye404.top 的标准流程

博客仓库位置：

- `/Users/fengye/workspace/blog/fengye404.github.io`

默认分支约定：

- 源码分支：`hexo-source`
- 发布分支：`gh-pages`

标准流程如下：

1. 确认当前在 `hexo-source` 分支上修改文章源码。
2. 编辑 `source/_posts/` 下对应文章；如果文章有配图，放在同名资源目录里。
3. 本地构建前先执行：
   - `npm run clean`
   - `npm run build`
4. 构建通过后执行：
   - `npm run deploy`
   这一步会把生成后的站点内容推到 `gh-pages`。
5. 回到源码仓库，检查变更：
   - `git status --short --branch`
6. 提交并推送 `hexo-source`：
   - `git add <相关文件>`
   - `git commit -m "..."`  
   - `git push origin hexo-source`
7. 最后确认两件事：
   - `hexo-source` 的源码提交已经推上去
   - `gh-pages` 已经由 deploy 更新

对用户回报时，尽量给出：

- 文章路径
- `hexo-source` commit id
- `gh-pages` commit id
- 线上文章地址

如果只是修改 skill，不需要走这套博客发布流程。
