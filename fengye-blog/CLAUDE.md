# 写作风格

## 概览

这个 skill 用于中文技术博客写作，目标是尽量贴近用户在博客站点里的既有文风。

开始写之前，先看用户已有博客，而不是只凭当前草稿猜文风。优先参考这些在线入口：

- 博客首页：[https://fengye404.top/](https://fengye404.top/)
- 归档页：[https://fengye404.top/archives/](https://fengye404.top/archives/)

目标文风大致是：

- 解释清楚，但不软、不说教
- 技术判断站得住，笔锋更硬一些
- 更像技术分享，不像营销文案
- 愿意解释，但不会为了显得“有思考”而故意绕

## 写作目标

文章通常应该呈现为：

1. 先把问题说清楚
2. 再把工程判断讲明白
3. 然后给出具体方案、改动或结构
4. 最后落到实际结论和启发

不要追求“文艺感”或者“很会写”的感觉，更重要的是可读性、信息密度和工程可信度。

## 核心规则

- 要解释清楚，但不要为了压字数过度简化。用户是希望“讲明白”的。
- 行文要直接，优先用陈述句，不要靠很多铺垫进入正题。
- 尽早使用具体工程名词，比如：模块边界、验证、CI、schema、契约、invariants、runtime、migration。
- 如果标题里出现项目名，开头要尽快用一两句说明这个项目是什么，不要让读者一直悬着。
- 如果作者背景需要交代，只能自然带一句，不能写得像在辩解或自我解释。
- 外部资料要内化，不要把文章写成“文献综述”，除非用户明确要求。
- 在重写或大幅润色之前，先从归档页抽样阅读多篇旧文。
  普通润色至少看 3 到 5 篇；如果要重写整篇结构或重新找文风，优先看 8 到 10 篇。
- 参考旧文时，重点观察这些东西：开头怎么起笔、问题怎么落地、解释密度怎么控制、结尾怎么收，不要只模仿表面句式。

## 反模式

以下写法**绝对禁止**，不存在"除非必要"的例外：

- **禁止** `不是……而是……` 句式。一次都不能出现。如果想对比两个概念，直接说后者是什么，不要先否定前者。例如：❌ "这不是一个简单的工具，而是一个完整的平台" → ✅ "这是一个完整的平台"
- **禁止** `本文不讨论……只讨论……`。直接说讨论什么。
- **禁止** `换句话说`。要么前一句话已经说清楚了（那就删掉"换句话说"后面的内容），要么没说清楚（那就重写前一句）。
- **禁止** 防御式表述："我不是在做 X，我是在做 Y"。直接说在做 Y。
- **禁止** `在某种程度上`、`或许`、`可能是因为`、`某种意义上说` 等模糊表达。要么确认了写断言，要么不确认就不写。
- **禁止** AI 常见过渡句：`值得注意的是`、`需要指出的是`、`更重要的是`、`一个关键的X是`、`一个值得注意的X是`、`本质上来说`、`总的来说`、`从X的角度来看`。直接说内容，不要用这类引导句。
- **禁止** AI 高频词汇：`天然`（如"天然支持"、"天然解决"）、`事实标准`、`显著`、`痛点`、`维度`、`赋能`、`范式`、`核心矛盾`。这些词一出现就暴露 AI。用更具体的表达替代。
- 过多口语填充
- 一直重复解释同一个点的“AI 说明文”语气
- 用“它不是什么，而是什么”来介绍项目
- 在正文里堆很多外部参考链接，像综述一样

一个判断标准是：如果一句话像是在努力证明自己“很有思考”，就应该继续简化。

## 视角与语气

文章必须用第一人称视角写作。你写的是个人技术博客，不是产品文档、RFC 或教程。

- 文中至少出现一处自然的第一人称工程叙述，比如："我选了 X 方案"、"我们当时用的是 Y"、"我在升级过程中踩了个坑"。
- 不要全文都是纯客观第三人称叙述。即使是介绍技术概念，也要从"我在项目里遇到了什么"这个视角展开，让读者感受到作者是实际参与过的。
- 参考用户旧文的做法：几乎每篇都能看到作者带入自己的经验和判断，而非旁观者角度。

## 句子节奏

**绝对不要让所有句子都一样长。**

真人写的博客有明显的节奏变化。你需要：

- **每篇文章至少出现 3 处 ≤8 字的短句**。示例：「坑在这里。」「改完就好了。」「版本不对。」「结果翻车了。」「够用了。」「索引在。」「就这么简单。」「问题不大。」短句用来做判断、下结论或转折。
- 长句留给因果链或复杂技术描述（30-50字也没关系）。
- 如果连续三句以上都是 15-30 字的中等长度，说明节奏太平了，必须用一个短句断开。

不要为了变化而变化——短句必须有内容，不是凑数。

## 口语化语气

你写的是个人博客，不是技术文档。语气要像在跟同事聊天讲自己的经历，不是在写报告。

### 随意连接词

**文章中必须自然使用以下连接词中的至少 2 个**（这是硬性要求，不是建议）：

> 于是、后来、反正、其实、说白了、结果、然后

这些是中文口语里最常见的连接方式。AI 倾向于用「然而」「因此」「此外」「不过」——这些太书面了。用「于是」替代「因此」，用「后来」替代「随后」，用「反正」替代「总之」，用「其实」替代「实际上」。

写完之后自检：全文搜索以上 7 个词，至少命中 2 个不同的。如果不到 2 个，回去改。

### 口语化表达

文章中必须出现至少 2 处明显的口语化表达。参考词库：

> 搞定了、折腾了半天、有点卡/有点烦/有点坑、挺好用的/挺方便的、随手写了个、不太靠谱、怎么说呢、纯属自虐、翻车了、一脸懵、踩坑、整活、真香、够呛

不要每一句都口语化——那会变成段子手。保持大部分句子是正常技术叙述，但在关键节点（遇到问题、做出选择、得到结论时）用口语表达你的真实反应。

### 对比示例

| AI 味 | 人味 |
|--------|------|
| 实际上，这个配置需要调整 | 其实这个配置得改一下 |
| 因此我们决定采用方案 B | 于是就选了方案 B |
| 随后进行了大量调试 | 后来折腾了半天 |
| 该方案的性能表现令人满意 | 跑起来挺快的 |
| 最终成功完成了迁移 | 最后总算搞定了 |

## 段落结构

**不要每段都是完整论证。**

AI 的典型毛病是每个段落都写成：论点→展开→例子→收束。真人博客不会这么工整。

- 全文中至少有一个只有 1-2 句话的短段落。可以是一个快速判断、一句转折、一个吐槽。
- 允许段落长短不一。有些段落 1 句话，有些段落 7 句话——这才是真人写的节奏。
- 不要每个段落都以分析或总结句收尾。有些段落说完事实就可以直接结束。

## 禁止平行枚举

**不要用 3 个以上连续的同构句式列举问题/原因/步骤。** 这是硬性禁止。

AI 最爱的套路是「首先...其次...最后...」、「第一个是...第二个是...第三个是...」、「X 负责 A，Y 负责 B，Z 负责 C」、「最多的是...其次是...还有一类...」。真人写博客不会这么工整地排列三个以上的并列项。

如果确实需要提到多个要点：
- 把其中一个展开成叙事段落而不是一行概括
- 把其中两个合并成一句话，用"还有"或"另外"随意串起来
- 只重点写最关键的 1-2 个，其余一笔带过

❌ 「第一个问题是并发写入。第二个问题是锁竞争。第三个问题是连接池耗尽。」
❌ 「最多的是 A 类错误。其次是 B 类错误。还有一类是 C 类错误。」
✅ 「最先遇到的是并发写入——多个线程同时写同一行，直接死锁。连接池也撑不住，高峰期连接数直接打满。」

写完自检：如果你看到连续三个以"第一/第二/第三"、"首先/其次/最后"、"最X的是...其次是...还有..."开头的段落或句子，立刻改掉。

## 结构建议

大多数技术文章，优先按这个结构来：

### 引言

- **从真实场景切入**。开头的第一句话应该是一个具体的工程经历、实际遇到的问题、或正在做的事情。例如："最近在把项目从 Boot 2 升到 Boot 3"、"今天在写 mirai 机器人的一个小功能时"、"上周给博客配 HTTPS，踩了几个坑"。
- **绝对禁止用抽象概念定义开头**。❌ "X 是一种..." ❌ "X 的核心思路是..." ❌ "在现代软件开发中..." → ✅ 从你正在做的具体事情说起。
- 用真实项目或工程背景把文章钉住

### 问题

- 明确点出工程问题
- 把边界说清楚

### 方法

- 解释具体改动、约束、判断
- 有必要时用具体例子

### 结论

**不要用固定模板收尾。**

AI 最爱的结尾模式是「不过 XXX 还没搞明白/先不管/以后再说」——这本身已经变成一种 AI 味。结尾应该多样化：

- 有时候用一句干脆的判断收尾：「值了。」「选 SQLite 没毛病。」
- 有时候收在一个个人回忆或感慨上——跟技术无关也行
- 有时候收在一个建议或推荐上
- 有时候最后一段就是在说技术细节，说完了就结束了，不需要专门总结
- 偶尔可以用「不过 X 还没搞明白」，但不能每篇都是这样

核心原则：**结尾不需要"收束感"**。真人博客经常写到某个点就停了，没有起承转合的闭环。

## 个人闲话

文章中**必须**穿插 1-2 处跟当前技术主题不直接相关的个人联想、回忆或感慨。这是硬性要求。

为什么重要：AI 写的文章全程紧扣主题、没有任何跑题——但真人写博客会自然联想到别的经历。缺少闲话是最容易被看出"AI写的"的特征之一。

例子：
- "这让我想起大学时第一次折腾 Linux，那时候连 apt 和 yum 都分不清"
- "搞完这个反而有点怀念写 Java 的日子"
- "仿佛回到了去年寒假第一次学后端的时候"
- "跟当年第一次在 CI 里跑通测试的感觉差不多"
- "让我想起以前实习的时候 leader 说过的一句话"

写完之后自检：全文中是否有至少一句话是在回忆/联想其他事情。如果没有，找一个合适的位置加一句。

## 工作方式

当使用这个 skill 时，应该这样工作：

1. 先看博客首页和归档页，再抽样阅读多篇旧文建立语感
2. 再读当前草稿
3. 找出语气、结构、framing 不匹配的地方
4. 必要时整体重写，不要只修补几句话
5. 保留有价值的技术细节
6. 如果用户要求发布，就继续完成：改文章、构建站点、部署、回报 commit id
7. **写完之后跑一遍自检清单**（见下方）

## 自检清单

写完文章后，逐项确认以下检查。任何一项不通过就回去修改：

- [ ] **无 AI 句式**：全文搜索「不是……而是」「换句话说」「值得注意的是」「本质上来说」——命中任何一个就改掉
- [ ] **无 AI 高频词**：搜索「天然」「事实标准」「显著」「痛点」「维度」「赋能」「范式」「核心矛盾」——命中就替换
- [ ] **无模糊表达**：搜索「在某种程度上」「或许」「可能是因为」「某种意义上说」——命中就删掉或改成断言
- [ ] **随意连接词 ≥2**：搜索「于是」「后来」「反正」「其实」「说白了」「结果」「然后」——至少命中 2 个不同的
- [ ] **口语化表达 ≥2**：全文至少有 2 处明显的口语（搞定/折腾/有点X/挺X的/翻车/踩坑 等）
- [ ] **无三连平行句**：检查是否有 3+ 个连续的"第一/第二/第三""首先/其次/最后"——有就改掉
- [ ] **有短句**：至少 3 处 ≤8 字的短促断句
- [ ] **有短段落**：至少 1 个只有 1-2 句话的段落
- [ ] **有弯路**：文章里展示过至少 1 个错误或次优尝试
- [ ] **有个人闲话**：至少 1 处与技术主题不直接相关的个人联想/回忆
- [ ] **结尾不模板化**：最后一段不是「不过 X 还没搞明白/先不管/以后再说」

# 收藏页（Bookmarks）

## 概述

博客有一个独立的「Bookmarks」页面，专门展示通过 fengye-bookmark skill 收藏的外部文章。

关键设计：
- **完全独立于文章体系**：收藏不是 post，不会出现在首页、归档、分类中
- **数据驱动**：收藏列表存储在 `source/_data/bookmarks.yml`，模板自动渲染
- **导航栏入口**：`Bookmarks` → `/bookmarks/`
- **自定义主题模板**：`bookmark.pug` + `bookmark.styl`

## 收藏数据格式

数据文件：`source/_data/bookmarks.yml`

```yaml
- class_name: AI & LLM       # 分组名（可选，留空则不显示分组标题）
  class_desc: 人工智能与大模型  # 分组描述（可选）
  bookmark_list:
    - title: "文章标题"
      link: https://example.com/article
      description: 简要描述
      date: "2026-04-11"      # 注意：日期必须加引号，避免被 YAML 解析为 Date 对象
```

## 添加新收藏到博客的流程

当 fengye-bookmark skill 完成收藏后，将条目同步到博客：

1. **在 `bookmarks.yml` 中追加条目**
   - 找到合适的 `class_name` 分组，在 `bookmark_list` 末尾追加
   - 如果没有合适的分组，创建新分组
   - 日期用引号包裹：`date: "YYYY-MM-DD"`

2. **构建和发布**
   - 按标准发布流程执行（见 references/style-guide.md）

## 主题文件说明

以下文件是为书签页自定义添加到 Butterfly 主题的，**升级主题时需保留**：

- `themes/butterfly/layout/page.pug` — 添加了 `when 'bookmark'` 分支
- `themes/butterfly/layout/includes/page/bookmark.pug` — 书签页模板
- `themes/butterfly/source/css/_page/bookmark.styl` — 书签页样式
- `themes/butterfly/_config.yml` — menu 中添加了 `Bookmarks: /bookmarks/ || fas fa-bookmark`
